# Project overview

このプロジェクトは Streamlit ベースのアプリです。  
主な目的は、修正時に Codex が以下を自動で回すことです。

- 原因調査
- 必要最小限の修正
- 回帰テスト追加
- pytest 実行
- Streamlit 起動確認
- 失敗時の再修正

---

# Execution environment

<!--
Codex に「どの環境でコマンド実行するか」を明示する。
今回の重要ポイントは WSL + .venv 固定。
-->

- このプロジェクトは **WSL 上で実行**する
- Python 実行環境は **`.venv/bin/python`** を使う
- Python コマンド実行時は `python` ではなく、現在の仮想環境を前提として扱う
- 依存関係は `.venv` に入っている前提で作業する

---

# Standard workflow

<!--
Codex が毎回やるべき標準フロー。
「修正して終わり」ではなく、テストと起動確認までやらせる。
-->

Codex はコード変更時に、必ず以下の順で作業すること。

1. まず根本原因を調査する
2. 必要最小限で修正する
3. 今回の不具合に対する回帰テストを追加または更新する
4. 以下を実行する
   - `python -m compileall app.py pages utils tests`
   - `python -m pytest -q`
5. テストが失敗した場合は、原因分析 → 修正 → 再実行を繰り返す
6. pytest 通過後、以下を実行する
   - `python -m streamlit run app.py --server.headless true`
7. Streamlit 起動に失敗した場合は、原因分析 → 修正 → 再実行を繰り返す
8. 最後に以下を報告する
   - 根本原因
   - 変更ファイル
   - 追加/更新したテスト
   - 未解決リスク
   - 人間が行う確認手順

# Testing policy

<!--
テスト追加の方針。
pytest は「書いたケースしか保証しない」ので、正常系・異常系・境界値を明示。
-->

バグ修正時は、必ず以下を意識してテストを追加すること。

- **正常系**: 想定どおりの入力で正しい結果になること
- **異常系**: 不正値・不足値・想定外入力で安全に失敗すること
- **境界値**: 0件、1件、None、空文字、重複などの境界条件を確認すること
- **回帰テスト**: 今回の不具合が再発しないことを保証すること

追加ルール:

- テストは `tests/` 配下に置く
- UI全体より、まずロジック単位・関数単位のテストを優先する
- テストしにくい場合は、最小限の関数分割は許可する
- 不要な大規模リファクタは禁止

---

# Pytest discovery rules

<!--
前回ハマった .venv 探索問題を再発させないためのルール。
-->

- pytest の探索対象は `tests/` のみに限定する
- compileall の対象は `app.py`, `pages/`, `utils/`, `tests/` のみに限定する
- `.venv`, `venv`, `venv313`, `__pycache__`, `.pytest_cache` は探索対象から除外する
- pytest 設定を壊す変更はしない
- 必要なら `pytest.ini` を維持・更新する

---

# Streamlit-specific notes

<!--
Streamlit は import 時点で重くなりやすいので注意点を明示。
-->

- Streamlit ページは import 時に重い処理を走らせないよう注意する
- 可能な限り、ロジック部分は純粋関数として切り出してテスト可能にする
- UI上のクリック操作そのものを自動で完全再現できない場合は、手動確認手順を明示すること
- `streamlit run` が即時クラッシュしないことを最低限確認すること

---

# Constraints

<!--
Codex にやってほしくないこと。
-->

- 不要な大規模リファクタは禁止
- DB スキーマ変更は禁止（明示依頼がある場合のみ可）
- 外部 API や認証方式の変更は禁止（明示依頼がある場合のみ可）
- 変更はレビューしやすい最小単位に留めること
- コメントやログを増やしすぎないこと
- デバッグコードを追加したら、不要なら最後に整理すること

-->

作業完了時は、以下の順で報告すること。

5. 実行結果

   - pytest

6. 未解決リスク
